Introduction

What is Naufrago!?

Naufrago! is a simple RSS reader that lets you view feeds with its images even not staying afloat, or put another way, without going online. This is because there are people, as the author, who does not always have a connection by hand, however, want to read articles / entries / posts with images that can accompany them. This is the leitmotif of the application.

What are the features?

The highlights are:

  • Feed categorization on a single category level tree.
  • Feed organization by drag & drop.
  • Online/offline reading (including images).
  • Smart category folding/unfolding.
  • Fullscreen mode for feed reading.
  • Marking feeds as important/permanent (recording).
  • Inclusion of favicons for feeds.
  • Application iconization on the system Tray.
  • Import/export feeds in OPML format.
  • Important and Unread folders.
  • Term searching capabilities.
  • Multi language (currently English, French, Polish, Italian, Catalan and Spanish).
  • Automatic update checker.

Which are the minimum requirements?

If you're reading this, you probably meet them. In terms of software:

  • GNU/Linux Operating System (given that some lib is not available for other OS)
  • Python
  • python-gtk2 (gtk2 bindings)
  • python-gobject (GObject bindings)
  • python-pysqlite2 (sqlite3 bindings)
  • python-feedparser (feedparser bindings)
  • python-webkit (webkit bindings)
In terms of hardware, anything that can move the app. The author has developed and uses it on a daily basis from a Netbook with Intel Atom 1.6 Ghz with 1GB of RAM.

Why reinvent the wheel?

If the client is so similar (with exceptions) to Liferea (at least in appearance), why reinventing the wheel? For many reasons. Mainly because the author could not find an RSS client that would work offline including images, Liferea developers do not accept feature requests (something quite respectable) and taking a look to third party source requires a dual effort comparing to create it (warning, personal opinion!) . Reinventing the wheel also means learning, improving and finding new ways of doing things may arise that otherwise would not have appeared.

Operation

How does offline mode works?

This mode ensures that you have access to the articles/entries/posts with its images even without an Internet conection, without interfering with other aspects in the functioning of the application. This means you can always leave it enabled, no matter you have or have not Internet access, and the application will still keep on getting new articles / entries / posts if it can. The only drawback is that leaving this mode always activated, it will consume more bandwidth and will take longer to get the updates, since the images will also be retrieved. NOTE: For the offline mode to work correctly, after you enable it on the preferences dialog, you must update all feeds whose images you want to view offline.

How does driven mode works?

This mode works by unfolding/folding these categories containing feeds with unread/read entries, respectively, making easy for the user the task of visual identification of what's left for reading. This also avoids unnecessary clics to the user.

I want to save important articles/entries/posts!

You can do this by checking the checkbox located at the beginning of the title of each article / entry / post.

I can not create subcategories

As mentioned in the features section of the application, the categorization allows only one level of categories.

I can not delete articles / entries / posts from feeds

It is true, they can not be erased. Instead, it was decided that the application manages its deletion automatically based on the number of articles / entries / posts configured in the preferences. So don't worry, the application will perform the cleaning for you.

Problems

The feed update stopping takes too long

Actually, this is inherent in how the cancel/stop feed updating works. For simplicity, we have defined an entire feed update as atomic operation, out of which you can proceed with the cancel/stop. Thus, during an update, when the cancel/stop is triggered, it will be effective when done with the last feed being updated at that time.

The date sorting of some feeds is incorrect

How this could be given that they usually are shown in order? By default and because feedparser is not foolproof at guessing all Internet RSS date formats, in this version (0.3) we have chose to sort feed entries on a first-come basis. So the sorting by the date column may have therefore the above problem.

The entries of some feeds don't show any content but titles (or even any title)

Just the same way there are feeds that does not publish dates for their entries, some others does not even provide content, but only the title (and sometimes even this is missing). Here the only way to go is to visit the site.

The application was closed unexpectedly!

Unfortunatelly, this happens sometimes and are unidenfied errors (abortions, segfaults, conection errors against X...). In order to assist you in the work of identifying potential errors, a log file has been enabled to help in this cases. The log is located at /usr/share/naufrago/naufrago.log for the Debian package and in the directory of decompression for the tarball. ANY HELP IS WELCOME if you want to contribute in this regard.
